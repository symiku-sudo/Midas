# Review（代码审查）

## 1) 必须流程
1. 明确评审范围（提交、分支、目录、目标行为）。
2. 按维度检查：正确性、边界条件、错误处理、配置、安全、可运维性。
3. 检查测试覆盖、回归风险与文档一致性。
4. 输出分级问题清单并给出定位信息。
5. 给出建议新增测试与残余风险结论。

## 2) 审查维度
- 逻辑正确性与边界条件。
- 错误处理与错误码一致性。
- 安全与敏感信息处理（日志、密钥、Cookie）。
- 配置与可运维性（配置驱动、默认值、回滚性）。
- 数据保护：是否误删/覆盖 `server/.tmp/backups/` 下 `.db` 备份（含 `*_latest.db`）。
- 测试覆盖与回归风险。
- 文档一致性：`README.md`、`server/README.md`、`server/API_CONTRACT.md`。

## 3) 结束门禁（任一不满足都不能结束）
- 高/中优先级问题未提供文件路径与行号，不能结束。
- 给出“可合并”结论但未说明测试覆盖与残余风险，不能结束。
- 声称“无问题”但未明确检查范围，不能结束。

## 4) 输出模板
1. 高优先级问题（必须修）。
2. 中优先级问题（建议修）。
3. 低优先级优化（可选）。
4. 建议新增测试。
5. 残余风险与合并建议。

## 5) 说明
- 每条问题尽量附文件路径与行号。
- 若未发现阻断问题，明确写“未发现阻断问题”，并说明剩余测试缺口。
