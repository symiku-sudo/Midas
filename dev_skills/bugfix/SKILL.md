# Bugfix（缺陷修复）

## 1) 必须流程
1. 明确复现步骤（输入、命令、期望、实际）。
2. 记录失败签名（报错、日志、接口返回、发生位置）。
3. 定位根因（文件 + 函数/类 + 触发条件）。
4. 实施最小修复，避免引入无关改动。
5. 补回归测试，覆盖本次失败路径。
6. 运行验证，确认失败签名消失且主流程可用。
7. 同步文档并输出交付摘要。

## 2) 结束门禁（任一不满足都不能结束）
- 无可复现步骤或失败签名记录，不能结束。
- 未给出根因链路与修复点映射，不能结束。
- 未补回归测试或无法证明失败签名消失，不能结束。
- 涉及配置/接口变化但文档未同步，不能结束。
- 涉及数据风险但无影响评估与复盘输出，不能结束。

## 3) 评分/召回/推荐类问题（强制流程）
- 适用范围：候选为空、候选过多、误报高、漏报高、排序不合理。
1. 抽取真实数据全集（当前库），输出样本与特征分布。
2. 建立人审基线：给出“应命中对/不应命中对”对照集（至少 10 对正样本；样本不足需说明）。
3. 全量打分并与人审对比，输出 Precision、Recall、TopK 命中。
4. 仅基于对比报告调权重/阈值，不凭直觉调参。
5. 调参后复跑同一对照集，展示前后指标变化。
6. 补回归测试，至少锁定 1 个“此前漏检”正样本。
7. 交付包含样本清单、阈值选择理由、残余误报风险。

硬约束：
- 未提供“人审 vs 程序”对照结果，不得结束任务。
- 仅改阈值且无特征解释，不得提交。

## 4) 数据风险事件复盘（命中时强制）
- 当问题触及“数据丢失/数据不可见/配置回滚”时，除修复外必须输出复盘：
1. 影响面：受影响数据范围与用户可见症状。
2. 时间线：触发时间与关键操作（使用绝对日期时间）。
3. 根因链路：触发条件 -> 代码/脚本 -> 最终表现。
4. 证据：日志、命令输出、文件路径、配置快照。
5. 防再发：最小可执行动作（代码、脚本、文档、告警）。

## 5) Midas 硬约束
- 保持错误码契约稳定且对外报错可读。
- 禁止向客户端暴露裸异常或 traceback。
- 涉及配置修复时同步更新示例配置与 README。
- 涉及数据或清理逻辑时默认保护 `server/.tmp/backups/` 下 `.db` 备份（含 `*_latest.db`）；如必须删除需用户明确授权。

## 6) 统一交付模板
- 根因摘要：一句话描述根因链路。
- 修复点清单：文件、函数与行为变化。
- 验证结果：失败签名消失证据 + 回归测试结果。
- 风险与回滚：残余风险、监控点与回滚方案。
- 文档同步：更新文件清单或“无文档变更”理由。
