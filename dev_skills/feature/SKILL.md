# Feature（功能开发）

## 1) 必须流程
1. 明确目标、范围和 DoD。
2. 给出实现计划：改动文件、验证方式、风险与回滚点。
3. 小步实现，优先复用现有服务与模型。
4. 为核心变更补至少 1 个测试（接口级或服务级）。
5. 执行验证并记录关键结果。
6. 同步文档并给出交付摘要。

## 2) 结束门禁（任一不满足都不能结束）
- 未提供验证证据（命令/结果/关键断言），不能结束。
- 未补对应测试或测试无法覆盖主路径，不能结束。
- 新增/变更 API 或响应字段但未更新 `server/API_CONTRACT.md`，不能结束。
- 变更使用方式但未同步 `server/README.md` 或根 `README.md`，不能结束。
- 新增可调参数但未更新 `config.example.yaml` 与 `config.yaml`，不能结束。

## 3) Midas 硬约束
- 不破坏统一响应协议：`ok/code/message/data/request_id`。
- 错误码语义保持可区分，不向客户端透传裸异常。
- 参数配置驱动，避免把可调项硬编码进业务逻辑。
- 涉及数据持久化或清理能力时，默认不得删除/覆盖 `server/.tmp/backups/` 下 `.db` 备份（含 `*_latest.db`）。

## 4) 评分/召回专项 DoD（命中此类改动时强制）
1. 提供离线评估报告，至少包含阈值扫描和前后指标对比。
2. 提供人审样本对照集，并说明阈值最终选择理由。
3. 增加回归样本测试，至少覆盖 1 个历史漏检样本和 1 个误报样本。

## 5) 统一交付模板
- 目标与 DoD：一句话说明完成标准。
- 变更文件：列出关键文件和改动点。
- 验证结果：列出执行命令、结果和关键断言。
- 风险与回滚：说明残余风险和可执行回滚路径。
- 文档同步：说明更新了哪些文档，或说明“无文档变更”理由。
